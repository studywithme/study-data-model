## 🎯 의미 전달이 왜 중요한가?

### 현실의 문제 상황

**시나리오**: 새로운 개발자가 팀에 합류했습니다.

```sql
-- 기존 테이블을 보고...
CREATE TABLE TBL_001 (
    COL_A VARCHAR(10),
    COL_B NUMBER,
    COL_C DATE,
    COL_D VARCHAR(1)
);
```

**새 개발자**: "이 테이블이 뭐하는 건가요?"
**기존 개발자**: "음... 잘 모르겠는데요. 코드를 봐야겠어요."

### 의미가 명확한 경우

```sql
-- 의미가 명확한 테이블
CREATE TABLE 고객 (
    고객ID VARCHAR(10) PRIMARY KEY,
    고객명 VARCHAR(50) NOT NULL,
    가입일자 DATE,
    활성상태 VARCHAR(1) CHECK (활성상태 IN ('Y', 'N'))
);
```

**새 개발자**: "아, 고객 정보를 저장하는 테이블이군요!"

## 🔍 의미 전달의 핵심 요소

### 1. 명명 규칙 (Naming Convention)

#### 나쁜 예시
```
테이블: T1, T2, TBL_A
컬럼: C1, C2, FLD_001
```

#### 좋은 예시
```
테이블: 고객, 주문, 상품
컬럼: 고객ID, 주문일자, 상품명
```

### 2. 비즈니스 용어 사용

#### 기술 중심 명명
```sql
CREATE TABLE USER_TRANSACTION_LOG (
    TRANS_ID NUMBER,
    USER_REF_ID VARCHAR(10),
    TRANS_TYPE_CD VARCHAR(5),
    PROC_DTTM TIMESTAMP
);
```

#### 비즈니스 중심 명명
```sql
CREATE TABLE 고객주문이력 (
    주문번호 NUMBER,
    고객ID VARCHAR(10),
    주문유형 VARCHAR(5),
    주문일시 TIMESTAMP
);
```

### 3. 관계의 명확한 표현

#### 모호한 관계
```
고객 --- 주문  (어떤 관계인지 불분명)
```

#### 명확한 관계
```
고객 --< 주문  (한 고객이 여러 주문을 할 수 있음)
     1:N
```

## 📊 의미 전달을 위한 모델링 기법

### 1. 엔티티명 작성 원칙

#### ✅ 좋은 엔티티명
- **구체적**: "사람" → "고객", "직원"
- **단수형**: "고객들" → "고객"
- **비즈니스 용어**: "USER" → "고객"

#### ❌ 피해야 할 엔티티명
- 추상적: "정보", "데이터", "마스터"
- 기술적: "테이블", "파일", "레코드"
- 모호한: "기타", "임시", "예비"

### 2. 속성명 작성 원칙

#### ✅ 좋은 속성명
```
고객ID, 고객명, 생년월일, 연락처
주문번호, 주문일자, 주문금액, 배송주소
```

#### ❌ 피해야 할 속성명
```
ID, NAME, DATE, AMT
COL1, COL2, FIELD_A, TEMP
```

### 3. 도메인 정의

각 속성이 가질 수 있는 값의 범위를 명확히 정의:

```
성별: 'M'(남성), 'F'(여성)
주문상태: '주문완료', '결제완료', '배송중', '배송완료', '취소'
등급: 'VIP', 'GOLD', 'SILVER', 'BRONZE'
```

## 🎭 실제 사례: 의미 전달의 실패와 성공

### 실패 사례: K 쇼핑몰

**문제 상황**:
```sql
CREATE TABLE CUST (
    C_ID VARCHAR(10),
    C_NM VARCHAR(50),
    C_TP VARCHAR(1),  -- 이게 뭘까?
    C_ST VARCHAR(1)   -- 이것도 뭘까?
);
```

**결과**:
- 개발자마다 다른 해석
- 잘못된 데이터 입력
- 유지보수 어려움

### 성공 사례: L 백화점

**개선된 모델**:
```sql
CREATE TABLE 고객 (
    고객ID VARCHAR(10) PRIMARY KEY,
    고객명 VARCHAR(50) NOT NULL,
    고객유형 VARCHAR(1) CHECK (고객유형 IN ('개인', '법인')),
    회원상태 VARCHAR(1) CHECK (회원상태 IN ('활성', '휴면', '탈퇴'))
);
```

**결과**:
- 직관적 이해
- 데이터 품질 향상
- 개발 생산성 증대

## 💡 의미 전달을 위한 실무 팁

### 1. 용어 사전 작성
```
용어: 고객ID
정의: 시스템에서 고객을 유일하게 식별하는 코드
형식: 영문 2자리 + 숫자 8자리 (예: CU12345678)
```

### 2. 비즈니스 규칙 문서화
```
규칙: 주문 취소 가능 기간
내용: 결제 완료 후 24시간 이내에만 취소 가능
모델 반영: 주문.주문일시, 주문.결제일시 속성으로 구현
```

### 3. 모델 리뷰 프로세스
- **비즈니스 담당자**: 업무 규칙 검증
- **개발자**: 구현 가능성 검토
- **DBA**: 성능 및 표준 검토
